================================================================================
           CONTEXTO DE INFRAESTRUCTURA Y SISTEMA - FORO HUB (2026)
================================================================================

Foro Hub es una solución backend API REST robusta para la gestión de comunidades digitales, desarrollada con Java y Spring Boot 3.x como parte del desafío de formación en el programa Alura/Oracle Next Education. El proyecto no solo replica el funcionamiento de un foro tradicional, sino que eleva el estándar mediante la implementación de seguridad avanzada, validación de contenidos con Inteligencia Artificial y persistencia corporativa en la nube.

--------------------------------------------------------------------------------
1. EL DESAFÍO Y DIFERENCIADORES TECNOLÓGICOS
--------------------------------------------------------------------------------

El Desafío:
    El objetivo principal fue construir una API escalable que gestionara tópicos de discusión, respuestas y perfiles de usuario, garantizando la integridad de los datos y una experiencia de usuario segura, fluida y eficiente.

Diferenciadores Tecnológicos:
    *   Seguridad de Grado Industrial:
        Implementación de Spring Security con autenticación JWT (Stateless), asegurando que cada interacción esté protegida y vinculada a roles específicos (ADMIN/USER).
    *   Moderación con Inteligencia Artificial Híbrida:
        Integración de modelos de lenguaje natural en tiempo real (Google Gemini y Mistral AI) en conjunto con filtros heurísticos locales para interceptar y bloquear contenidos ofensivos antes de que impacten la base de datos.
    *   Resiliencia de Datos (Soft Delete):
        Uso de Borrado Lógico mediante la bandera 'activo' (Boolean) y la anotación de Hibernate (@SQLRestriction), lo que permite omitir automáticamente registros inactivos en las consultas y mantener la trazabilidad sin pérdida física de registros.
    *   Observabilidad Avanzada:
        Monitoreo detallado del consumo de las APIs de IA (tokens y latencia exacta) mediante el patrón de diseño Decorator, permitiendo un control analítico sobre el rendimiento y los costos de operación.
    *   Diseño de Código Limpio (SOLID):
        Arquitectura basada en principios de diseño limpio, utilizando DTOs para la transferencia segura de datos y una estricta separación de responsabilidades entre controladores, servicios y repositorios.

3. MODELO DE DATOS Y ENTIDADES (PERSISTENCIA)
--------------------------------------------------------------------------------

Todas las entidades manejan Borrado Lógico mediante la propiedad 'activo' y la directiva '@SQLRestriction'.

Entidades Estructurales:
    *   Usuario:
        - Atributos: Long id, String nombre, String email, String password, Boolean activo.
        - Relaciones: Una lista de perfiles asociados (List<Perfil> perfiles).
    *   Perfil:
        - Atributos: Long id, PerfilNombre nombre.
        - Enumeración PerfilNombre: [ROLE_ADMIN, ROLE_USER]
    *   Curso:
        - Atributos: Long id, String nombre, String categoria, Boolean activo.
    *   Topico:
        - Atributos: Long id, String titulo, String mensaje, LocalDateTime fechaCreacion, StatusTopico statusTopico, Boolean activo.
        - Relaciones: Pertenece a 1 Usuario (autor), pertenece a 1 Curso (curso), y contiene una lista de aportes (List<Respuesta> respuestas).
        - Enumeración StatusTopico: [NO_RESPONDIDO, NO_SOLUCIONADO, SOLUCIONADO, CERRADO]
    *   Respuesta:
        - Atributos: Long id, String mensaje, LocalDateTime fechaCreacion, Boolean solucion, Boolean activo.
        - Relaciones: Vinculada de forma obligatoria a 1 Usuario (autor) y a 1 Topico (tópico padre).

Relaciones del Ecosistema:
    *   Un Usuario posee 1 o muchos Perfiles.
    *   Un Usuario puede autorizar de 0 a muchos Tópicos o Respuestas.
    *   Un Tópico pertenece estrictamente a 1 Curso.
    *   Un Tópico aloja de 0 a muchas Respuestas en su hilo de discusión.

4. FLUJOS OPERATIVOS Y LÓGICA DE NEGOCIO
--------------------------------------------------------------------------------

A) Proceso de Autenticación JWT (Stateless):
    1. El cliente envía sus credenciales en formato JSON vía POST al endpoint '/login'.
    2. El 'AuthenticationManager' coordina la solicitud y delega en 'UserDetailsService' la búsqueda del registro a través de 'UsuarioRepository'.
    3. Se valida el texto plano de la contraseña contra el hash almacenado mediante 'PasswordEncoder' (BCrypt).
    4. En caso de éxito, el 'JWTProvider' firma digitalmente un token criptográfico y retorna un HTTP 200 OK con el token estructurado. Si falla, se lanza una 'BadCredentialsException' devolviendo un HTTP 401 Unauthorized.

B) Flujo de Creación y Consulta de Tópicos:
    1. El cliente inicia una petición POST ('/topicos' o '/respuestas') adjuntando en la cabecera el parámetro 'Authorization: Bearer <JWT>'.
    2. El servicio correspondiente verifica de forma explícita la existencia previa de los IDs vinculados (autor, curso o tópico).
    3. Para solicitudes de listados globales ('GET /topicos?page=0'), el backend procesa los datos de forma paginada ('Pageable') invocando métodos de filtrado activo como 'findByActivoTrue()' para restringir registros ocultos.

C) Regla de Negocio Estratégica para Soluciones (Marcado de Solución):
    Cuando una respuesta es seleccionada y marcada como la solución definitiva de un hilo de discusión a través de 'RespuestaService', se ejecuta de forma atómica:
    1. Se desmarcan de manera automática todas las respuestas previas ligadas a ese mismo tópico específico (forzando su estado a 'solucion = false').
    2. Se sobrescribe el atributo a 'solucion = true' exclusivamente en la entidad de respuesta seleccionada.
    3. El parámetro 'statusTopico' del Tópico padre se actualiza obligatoriamente al estado transaccional 'SOLUCIONADO'.
    4. El gestor de persistencia realiza el commit definitivo de la transacción en la base de datos.

5. SISTEMA DE MODERACIÓN INTELIGENTE MULTICAPA (AI FAILOVER)
--------------------------------------------------------------------------------

Para salvaguardar la integridad de la comunidad ante interacciones ofensivas o spam, el sistema intercepta las solicitudes de creación mediante un pipeline en cascada con alta disponibilidad (High Availability):

Capas de Decision:
    *   Capa Primaria (Cloud Google Gemini):
        Utiliza el modelo 'Gemini-3-Flash-Preview' en la nube para procesar análisis semánticos complejos, evaluaciones de contexto, sentimiento y detección de sarcasmo.
    *   Capa Secundaria / Failover (Cloud Mistral AI):
        Si la API de Gemini experimenta caídas, latencia excesiva o cuotas de tokens agotadas, el sistema conmuta en milisegundos hacia la API de 'Mistral Large' para dar continuidad al proceso.
    *   Capa de Seguridad Local (Heurístico Offline):
        Si la infraestructura pierde conectividad externa o ambos proveedores de IA fallan, se activa un motor de respaldo nativo en Java que calcula una puntuación de riesgo determinista utilizando diccionarios de palabras clave y métricas de SPAM locales.

Manejo de Errores en Moderación:
    Si cualquiera de las capas de control determina una vulneración ('esContenidoOfensivo == true'), el sistema aborta la operación, lanza inmediatamente una 'ValidacionException', revierte cualquier inserción en base de datos (rollback) y retorna un código de respuesta HTTP 400 Bad Request acompañado de un mensaje controlado de bloqueo.

6. OBSERVABILIDAD Y MONITOREO DE IA
--------------------------------------------------------------------------------

El consumo de los modelos de IA se fiscaliza rigurosamente mediante el patrón de diseño Decorator implementado en la clase 'LoggingChatModelDecorator'. Este componente envuelve el cliente de chat registrando métricas directo en los logs de la infraestructura:
    *   Nombre y metadatos del modelo de lenguaje en ejecución.
    *   Cantidad exacta de tokens de entrada (Input Tokens): Típicamente ~174 tokens que integran el cuerpo del mensaje y el prompt con las reglas de moderación.
    *   Cantidad exacta de tokens de salida (Output Tokens): Típicamente ~3 tokens conteniendo la estructura del veredicto final (APROBADO/RECHAZADO).
    *   Tiempo de respuesta exacto y latencia de la petición HTTP externa.

8. ESPECIFICACIÓN DE ENDPOINTS Y ESTRUCTURA DE DTOS
--------------------------------------------------------------------------------

A) Módulo de Autenticación:
    *   `POST /login` -> Acceso PÚBLICO. Autentica identidades y genera la firma del token.
        - Request (DatosAutenticacionUsuario): [String email, String password] -> Campos validados con @NotBlank.
        - Response (DatosJWTToken): [String jsonWebToken] -> Estado HTTP 200 OK.

B) Módulo de Tópicos:
    *   `POST /topicos` -> Requiere JWT. Registra un nuevo hilo de discusión (Retorna HTTP 201 Created).
        - Request (DatosRegistroTopico): [String titulo, String mensaje, Long autorId, Long cursoId] -> Validados con @NotBlank y @NotNull.
        - Response (DatosRespuestaTopico): [Long id, String titulo, String mensaje, LocalDateTime fechaCreacion, StatusTopico statusTopico, String nombreAutor, String nombreCurso]
    *   `GET /topicos` -> Requiere JWT. Recupera un listado global paginado.
        - Parámetros nativos de ordenamiento: `?page=0&size=10&sort=fechaCreacion,desc`
        - Response: Page<DatosRespuestaTopico> (Excluye automáticamente registros inactivos).
    *   `GET /topicos/{id}` -> Requiere JWT. Detalle de un registro único.
        - Response: DatosRespuestaTopico estructurado. Si el ID no existe o su estado 'activo' es falso, retorna un HTTP 404 Not Found.

C) Módulo de Respuestas:
    *   `POST /respuestas` -> Requiere JWT. Evalúa el mensaje mediante el flujo de IA antes de persistir (Retorna HTTP 201 Created).
        - Request (DatosRegistroRespuesta): [String mensaje, Long topicoId, Long autorId] -> Validados con @NotBlank y @NotNull.
        - Response (DatosRetornoRespuesta): [Long id, String mensaje, LocalDateTime fechaCreacion, String nombreAutor, Boolean solucion]
    *   `PUT /respuestas/{id}/solucion` -> Requiere JWT. Modifica el estado del aporte para resolver el debate (Retorna HTTP 200 OK).
        - Acción: Ejecuta la regla de negocio para fijar la solución, limpiar registros anteriores del hilo y actualizar el tópico padre a SOLUCIONADO.
        - Response: DatosRetornoRespuesta con propiedades actualizadas.
